**版本定调**：评审 C 指出早期草稿把首要矛盾排错了——v1.0 最大的正式版阻断不是「流程不够顺」，而是**「平台稿」根本不是能拿去平台发布的作品**（小红书稿没有图片、公众号稿是纯文本、X 无法表达 Thread 且字符计数不按官方加权规则）。本 PRD 据此确立：以「创作项目 + 可选通用稿 + 类型化平台作品」为中心概念，彻底取消「包装」，readiness 下沉到平台级，并明确采用**破坏式数据重构**（产品未交付用户，现有数据均为测试数据，不做无损升级）。

## 1. 背景与定位

Muse 是面向自媒体创作者的本地优先、单用户「一站式创作工厂」。feat-001 ~ feat-028 完成了模块闭环：内容、版本、审阅、平台稿、发布记录、复盘经验可保存且相互溯源。三轮独立评审的递进结论：

1. 审计 A/B：**模块闭环已成，用户心智闭环未成**——状态矛盾、AI 幻觉、承诺未兑现。
2. 评审 C：**心智闭环之下还有更硬的地基问题**——平台产物不符合真实平台的作品结构，「修补流程」无法解决「产物不可用」。

v1.0 的使命据此修订：

> **让一个创作项目真正产出可直接发布的平台作品。** 判据只有一条：从 Muse 拿到结果后，创作者**不需要重新组织内容、重新配图、重新排版**，即可进入平台后台完成发布。

## 2. 目标与非目标

### 2.1 目标（v1.0 成功标准）

1. **产物可发布**：四种平台作品类型（X 单条 / X Thread / 小红书图文笔记 / 公众号图文文章）各自具备平台原生结构、专属编辑器、平台效果预览、发布检查与手动发布助手；通过「三平台真实后台发布前一步」演练。
2. **状态可信**：全站唯一的派生阶段语言；readiness 按平台独立，一个平台的缺口不阻断另一个平台发布。
3. **AI 可信**：所有 AI 写入/派生路径不得静默引入无来源的事实断言；所有 AI 操作 2 秒内有反馈、可取消、失败可恢复。
4. **承诺兑现**：三个创建起点走通不中断；发布记录可补录；复盘按平台记录并回流。

### 2.2 非目标（明确不做）

- **真实平台发布 API / 自动直发**。手动发布助手是 v1.0 的正式方案（三方评审一致）。
- 小红书**视频笔记**、X Premium **长帖**（进 v1.x，避免 v1.0 同时承担视频生产与账号能力判断）。
- 平台表现数据自动抓取；多用户 / 协作 / 云同步；热点榜单等外部内容源。
- 复杂 CRM 式账号运营（只做克制的「创作档案」）。
- **旧测试数据的无损升级**（见 FR-0.1，破坏式重构）。
- 全链路流式输出降为 v1.x（v1.0 以「2 秒反馈 + 可取消 + 失败可恢复」为底线，见 FR-4.3）。

## 3. 核心概念模型

### 3.1 概念定义

| 概念 | 定义 | 替代/取消的旧概念 |
|---|---|---|
| **创作项目**（creation） | 一次创作的容器：内部工作标题、Brief、目标平台集合、阶段。内部工作标题**不等同于任何平台的发布标题** | `articles` 的项目属性 |
| **通用稿**（source document） | 可选的跨平台内容母版，带不可变修订（source revisions）。多平台分发时推荐使用；单平台直写时**不存在** | `articles` 的正文属性 + `article_versions` |
| **平台作品**（platform output） | 某平台某格式的真实作品，结构由类型化 payload 定义，带独立修订链、独立 readiness | `platform_variants`（通用表单）|
| **图片资产池**（assets） | 项目级图片库。用户只在**具体平台作品内**使用它（选图、排序、设首图/封面、裁剪、alt 文本） | `packagings` 的图片职责 |
| **发布记录**（publication） | 对某平台作品修订的**不可变快照** + 可编辑元数据（链接/时间/备注） | `publish_tasks`/`publish_results` |
| ~~包装~~ | **概念删除**。标题候选、摘要、封面提示词、图文卡片不再作为跨平台共享产物存在 | `packagings` 全部 |

### 3.2 四种平台作品类型（v1.0 全部支持）

payload 采用 Zod 判别联合 + `schema_version`，公共可查询字段留关系列，平台专属结构进类型化 payload。

| 类型 | 平台原生结构（payload 要点） | 就绪硬条件（发布检查阻断项） |
|---|---|---|
| `x_single_post` | `text`、`media[]`（≤4，图/GIF/视频引用）、链接 | 加权字符 ≤280（官方 `twitter-text` 算法：CJK/emoji 加权、URL 按 t.co 固定长度）；**无对外标题/摘要字段** |
| `x_thread` | 有序 `posts[]`，每条独立 `text` + `media[]`（≤4） | 每条独立通过加权字符校验；至少 2 条；逐条可复制 |
| `xiaohongshu_image_note` | `title`、`body`、`topics[]`、**有序 `images[]`（1–18 张）**、首图标记 | **无图片 = 不就绪**；标题 ≤20 字；图片可排序、可打包下载 |
| `wechat_article` | `title`、`author`、`digest`（摘要）、**HTML 正文**、`cover`（封面资产）、正文内图片、原文链接 | 封面缺失 = 不就绪；HTML 导出可直接带入公众号编辑器 |

每种类型各自拥有：**专属编辑器 → 平台效果预览 → 发布前检查 → 手动发布助手 → 导出/复制方式**（详见 FR-1.x）。

### 3.3 目标数据模型（逻辑层）

```text
creations                          创作项目（工作标题、brief、目标平台、阶段派生依据）
source_documents                   可选通用稿（0..1 per creation）
source_revisions                   不可变通用稿修订

platform_outputs                   平台作品（platform, format, active_revision_id,
                                   source_revision_id?, rules_version）
platform_output_revisions          作品修订（payload_json: Zod 判别联合, schema_version）
output_assets                      修订 ↔ 资产关联（role: 首图/封面/正文图/post附件,
                                   order, alt_text, crop_json）

publications                       发布记录（output_revision_id 不可变快照,
                                   url / published_at / note 可编辑）
performance_snapshots              表现快照（platform 指标模板, 口径时间戳, 多快照）

reviews / review_findings          审阅：多态挂载 source_revision 或 output_revision
                                   （单平台直写路径的检查直接作用于作品文本）
topics / materials / collections / retro_notes / app_settings   保留，语义不变
```

**平台规则注册表**：`platforms.ts` 的写死常量重构为规则注册表——每平台每格式的校验规则携带 `rules_version`、来源 URL、核对日期；作品记录生成时的 `rules_version`，规则更新后可提示「按新规则重检」。

## 4. 目标用户与核心旅程

**画像**：单人、高频（每周 3~5 篇）、自负盈亏、多平台运营（公众号 + X + 小红书），最大恐惧是内容翻车（事实错误、编造数据被揪出）。

**v1.0 基准旅程**：

```text
起点（想法 / 资料篮子 / 经验）
  → 选择目标平台（单平台 or 多平台）
  ├─ 单平台 → 直接进入该平台作品编辑器（无通用稿步骤）
  └─ 多平台 → 推荐先写通用稿 → 派生各平台作品 → 逐平台深度调整
  → 写作与检查（审阅/事实核对作用于当前编辑对象）
  → 各平台作品独立：编辑 → 平台预览 → 发布检查（独立 readiness）
  → 就绪的平台先发：手动发布助手（按平台裁剪）→ 回填链接（可后补）
  → 按平台记录表现（指标模板 + 口径）
  → 经验回流为新创作起点 ↺
```

「从一份现成平台作品适配到另一个平台」是一等操作，无需先补造通用稿。

## 6. 需求详述

优先级：P0 = 正式版阻断；P1 = 正式版应有；P2 = 可随 v1.0.x。
来源：[A-*] 审计 A；[B-*] 审计 B；[C-*] 评审 C。

### 主题〇：破坏式数据重构（Foundation）

#### FR-0.1 破坏式 v1 数据重构 — P0 [C-4.6]

**背景**：产品未交付任何真实用户，当前数据库全部为测试数据。早期草稿的「无损升级」要求与此冲突，且会留下长期兼容负担。

**需求**：
- 按 §3.3 重建数据模型；允许重建本地测试数据库（提供一次性重置脚本 + 种子数据脚本，替代历史数据迁移）。
- **删除**：`articles.status` 面向用户的一切渲染与兼容分支、`packagings` 表及其全部 UI/action、旧 `platform_variants` 表结构。
- **不迁移**旧包装记录与旧平台稿（语义无法可靠映射到类型化作品）；素材/选题/复盘经验等语义未变的数据可选保留。
- JSON 全量导出覆盖新模型；不承诺旧导出文件可导入。

**验收**：全新初始化数据库跑通全部动线；代码库中无 `packagings`、无面向 UI 的 `articles.status` 引用；`./init.sh` 全绿。

#### FR-0.2 平台规则注册表 — P0 [C-证据§3]

**需求**：平台校验规则（字符权重、图片数量/比例、标题长度、封面要求等）集中于规则注册表，每条规则携带来源 URL、核对日期、`rules_version`；X 字符计数采用官方 `twitter-text` 加权算法（CJK/emoji 权重、URL 按 t.co 固定长度），**禁止用 `string.length` 或普通字数判断可发布性**。作品修订记录其生成时的 `rules_version`。

**验收**：含中文+emoji+URL 的 X 文本，计数结果与 X 官方 composer 一致（抽样 ≥10 例）；规则文件中每条规则可见来源与日期。

### 主题一：平台作品（Platform Outputs — v1.0 核心）

> 通用验收（适用于 FR-1.1~1.4）：作品有独立 readiness 与发布检查；预览接近平台真实展示（首屏截断、字符计数、标签/话题渲染）；发布助手按平台裁剪；发布时冻结修订快照。

#### FR-1.1 X 作品：单条 + Thread — P0 [C-4.2][A-14]

**需求**：
- `x_single_post`：文本 + ≤4 媒体；编辑器实时显示加权字符余量；**不存在对外标题/摘要字段**（内部备注仅作 Muse 侧信息，明确标注不会发布）。
- `x_thread`：一等作品类型（不再是 P2）。分条编辑器：增删、排序、每条独立文本与媒体、每条独立字符校验；预览按 Thread 串联样式展示。
- 发布助手：单条一键复制；Thread 逐条复制（带 1/N 进度指示，复制完一条自动高亮下一条）。

**验收**：超 280 加权字符的条目红色阻断；Thread 三条各自可复制且顺序正确；含 URL 的文本计数按 t.co 长度。

#### FR-1.2 小红书作品：图文笔记 — P0 [C-4.2][C-证据§3]

**需求**：
- `xiaohongshu_image_note`：标题（≤20 字）、正文、话题、**有序图片 1–18 张 + 首图标记**。图片从项目资产池选择，支持排序、设首图、alt 文本。
- **无图片时不得判定就绪**——这是发布检查的硬阻断，不是提醒。
- 预览：模拟笔记样式（首图 + 标题 + 正文折叠 + 话题）。
- 发布助手：复制标题、复制正文+话题、**图片按顺序打包下载**（文件名带序号）。

**验收**：0 图作品显示「缺少图片，不可发布」且无法标记已发布；5 图作品调序后下载包内序号与预览一致。

#### FR-1.3 微信公众号作品：图文文章 — P0 [C-4.2][C-证据§3]

**需求**：
- `wechat_article`：标题、作者、摘要（digest）、**HTML 正文**（沿用 Tiptap 富文本）、封面资产、正文内图片、原文链接。
- 封面缺失 = 不就绪（硬阻断）；摘要缺失 = 提醒。
- 预览：文章样式（封面 + 标题 + 摘要 + 正文渲染）。
- 发布助手：导出 HTML（正文图片以可搬运方式打包/内联）、单独下载封面与正文图片、复制标题/摘要。导出物能实际带入公众号后台编辑流程。

**验收**：导出的 HTML 粘贴进公众号草稿编辑器后结构不散架（标题层级、图片占位、加粗/引用保留）；缺封面时不可标记发布。

#### FR-1.4 作品独立编辑 + 「编辑 / 预览 / 发布检查」三视图 — P0 [C-4.2][B-P1-3]

**需求**：每种作品类型的编辑器为其结构专属（Thread 分条、小红书选图排序、公众号富文本），统一提供三个视图：编辑（结构化表单/编辑器）、预览（平台效果）、发布检查（按规则注册表出具的 checklist：阻断项红色、提醒黄色）。

**验收**：三平台四类型各自的检查清单条目与 §3.2 硬条件一致；检查通过前「标记已发布」不可用（允许显式「带风险发布」需二次确认并记录）。

#### FR-1.5 图片资产池（替代包装的图片职责） — P0 [C-4.1]

**需求**：项目级图片上传/管理池保留；「封面/首图/正文图/Post 附件」等角色**只在具体作品内指定**（output_assets：role/order/alt/crop）。全局 `cover` 概念删除。可选保留 AI 配图提示词生成，作为资产池的辅助工具（生成提示词文案，不自动生图）。

**验收**：同一张图片可在公众号作品中作封面、在小红书作品中作第 3 图，互不影响；删除包装面板后无功能缺失投诉路径（标题/摘要在作品内、图片在资产池+作品内）。

### 主题二：创作动线（Flow）

#### FR-2.1 创建即选平台：单平台直写 / 多平台通用稿 — P0 [C-4.3]

**需求**：
- 创建向导（三起点保留）在方向确认后询问「这次准备发布到哪里」（默认值来自创作档案 FR-7.1，可改）。
- **单平台** → 跳过通用稿，直接创建该平台作品并进入其专属编辑器；审阅/事实检查直接作用于作品文本（reviews 多态挂载）。
- **多平台** → 推荐通用稿路径：写通用稿 → 一键派生全部目标平台作品（逐个执行、可单独取消）→ 逐平台调整。
- 「适配到另一个平台」：任何现成平台作品可直接派生其他平台作品，无需先补造通用稿（溯源记录 output→output 派生关系）。

**验收**：只发一条 X 的用户从想法到可复制的单条作品 ≤3 步（方向→Brief→作品编辑器）；多平台项目一键得到 N 份类型化作品；作品间适配的溯源链可见。

#### FR-2.2 空编辑器双入口 NextAction — P0 [A-4]

**需求**：通用稿或单平台作品为空时，NextAction 并列「自己开始写」与「按 Brief 生成初稿」（预览→确认→保存修订契约不变）。AI 初稿入口不得深于一次点击。

**验收**：三种起点创建后首屏均可一键触达 AI 初稿。

#### FR-2.3 「从资料开始」创作篮子 — P1 [B-P0-2]

**需求**：从创建向导进入素材库保持创建会话；素材/集合可勾选入篮，固定显示「已选 N 条 · 继续生成方向」；素材详情/集合页提供「用这些资料开始创作」；确认后自动关联为项目相关素材。

**验收**：勾选 3 条资料 → 生成方向 → 创建后资料面板「相关素材（3）」就位。

#### FR-2.4 方向候选「换一批」 — P2 [A-15]

**需求**：方向候选页支持保留输入重新生成（可携带「避开上一批角度」提示）。

#### FR-2.5 通用稿更新的差异更新与手工编辑保护 — P1 [C-6][B-P1-3]

**需求**：通用稿产生新修订后，平台作品标记「可能过期」；更新走「查看差异 → 确认更新」，**禁止静默覆盖**；若作品含手工编辑，差异视图明确展示将被影响的段落，用户可逐段选择保留。

**验收**：改通用稿 → 小红书作品的手工改动在更新后按用户选择保留；无静默覆盖路径。

### 主题三：状态一致性（Trust）

#### FR-3.1 全站统一派生阶段 + 平台级 readiness — P0 [A-6][B-P0-1][C-4.4]

**需求**：
- 面向用户只展示一套派生阶段：`方向 / 写作 / 制作平台作品 / 发布 / 复盘`（原「检查/发布准备」并入写作与作品制作，因检查跟随编辑对象）。覆盖：首页、创作列表（含筛选）、项目页头、作品页、⌘K、发布记录。
- **readiness 下沉到平台作品级**：每个作品独立就绪状态；项目聚合展示「1/3 个平台已就绪」；一个平台缺口不阻断另一平台发布。
- 发布时冻结该作品修订快照（publications 指向不可变 revision）。

**验收**：构造「X 就绪、小红书缺图、公众号缺封面」项目：X 可独立发布；聚合显示 1/3；六处 UI 阶段一致。

#### FR-3.2 空项目状态静默 — P0 [A-5]

**需求**：无实质内容的项目不产生任何「待更新/需对齐」徽章与待办；「全部待办」同步排除。

#### FR-3.3 检查的诚实语义 — P0 [A-7]

**需求**：存在未处理审阅发现时，对应编辑对象显示「N 条待处理」中性态而非绿✓；全部处理或显式「带风险跳过」（记录原因）后才转绿。

### 主题四：AI 可信（AI Reliability）

#### FR-4.1 全路径事实安全 — P0 [A-1][C-4.5]

**背景**：实测 AI 润色为「缺数据支撑」的 fact finding 编造「麦肯锡2023研究…35%」写入预览。早期草稿只约束 fact 润色；评审 C 指出初稿生成、选区改写、通用稿→作品派生、作品增量更新、标题/摘要生成同样可能编造。

**需求**（正式版规则）：
> **所有 AI 覆盖或派生内容，只要引入原文/关联资料中不存在的数字、机构、人物、报告、时间或事实断言，必须能定位来源，或显式标记「待确认」，不得静默进入可发布作品。**
- 实现：提示词层约束 + 生成后校验（新增数字/专名与原文+资料比对）；命中时高亮标注并给两条出路——「去资料库找证据插入引用」或「插入 `【待确认】` 占位」。
- 含未处理 `【待确认】` 标记的作品在发布检查中为阻断项。
- 覆盖路径：初稿、选区改写、润色、通用稿→作品派生、作品间适配、差异更新、标题/摘要/话题生成。

**验收**：一组对抗性测试（≥8 用例：诱导补数据、诱导引研究、跨路径各至少 1 例）全部不出现静默编造；`【待确认】` 未清除时无法标记发布。

#### FR-4.2 审阅结果即时呈现 — P0（缺陷修复）[A-2]

**需求**：审阅/事实检查完成后，建议列表、阶段、待办计数即时更新，无需刷新；0 发现时显示「未发现需要修改的问题」。

#### FR-4.3 AI 反馈底线：2 秒反馈、可取消、失败可恢复 — P0 [A-3][C-6]

**需求**（流式全面化降为 v1.x）：所有 AI 操作 2 秒内出现可见进度反馈；执行中可取消；失败/超时给出「重试 / 查看兜底稿（标注来源）」选择，不静默以模板稿填充；默认超时 20s 可配置。

**验收**：断网下任意 AI 操作 5 秒内出现失败反馈与重试；取消后 UI 状态干净。

#### FR-4.4 审阅决策队列 — P1 [B-P1-2]

**需求**：审阅结果分层摘要（阻断项 / 重要改进 / 可选润色），默认展开阻断项；队列化逐条处理（预览 diff → 接受/忽略 → 下一条）；批量忽略可选润色；处理进度徽标；「对已修改内容重新检查」。

**验收**：6 条建议全处理的点击次数 ≤ 旧交互 60%。

### 主题五：发布与记录（Publish）

#### FR-5.1 发布记录可编辑 — P0 [B-P0-3]

**需求**：发布记录内联补充/修改链接、发布时间、备注；复盘完成后仍可修正发布元数据；关联的作品修订快照不可变。

**验收**：空链接发布 → 补链接 → 复盘与经验卡溯源立即可用。

#### FR-5.2 平台表现指标模板与口径 — P1 [A-11][B-P1-5][C-6]

**需求**：按平台定义指标模板（公众号：阅读/在看/点赞/转发/留言；X：浏览/点赞/回复/转发/书签；小红书：浏览/点赞/**收藏**/评论/转发）；每次录入自动记录「数据截至发布后 N 天」；同一发布支持多时间点快照；发布记录与经验卡提供「更新数据」。

**验收**：小红书复盘表单含收藏；同一发布存 24h/7d 两快照且展示口径。

### 主题六：复盘（Learning Loop）

#### FR-6.1 假设登记与对照 — P1 [A-10]

**需求**：Brief 增加可选「本次想验证什么」（schema 随 M1 建表，UI 可在 M2 交付）；标记发布时可补充假设；复盘向导自动带出登记假设逐条判定「支持/未支持/数据不足」；无登记假设时改问「这次有什么意外发现」。

**验收**：登记假设的项目复盘自动对照；未登记的不再出现无源之问。

#### FR-6.2 摘要矛盾提示 — P2 [A-11]

**需求**：复盘摘要与数据明显冲突时（如评论 0 却判「引发讨论」获支持）黄色提示确认，不阻断保存。

### 主题七：素材与档案（Materials & Profile）

#### FR-7.1 账号创作档案（最小版） — P1 [B-P2-2][C-6]

**需求**：设置内新增创作档案：账号定位、目标读者、内容支柱（≤5）、语气人称、禁用表达、主要平台。作为创建向导默认值（替代来历不明的默认勾选）、审阅（禁用表达检查）与平台适配的默认上下文。全部可选。

**验收**：填档案后新项目 Brief 默认值与档案一致；正文命中禁用表达时审阅提示。

#### FR-7.2 导入即整理、用户语言化 — P1 [A-13][B-P2-1]

**需求**：素材导入后自动整理（失败才显示重试）；文案去实现语言（「清洗为语料块」→「整理为可引用片段」，隐藏「FTS5」）。

**验收**：导入 URL 后零点击即可被搜索命中；界面无实现术语。

### 主题八：响应式与可访问性

#### FR-8.1 窄屏辅助面板抽屉化 — P1 [B-P1-1]

**需求**：<768px 时审阅/版本/资料面板与作品检查清单改为底部抽屉，保留正文滚动位置；顶部固定当前步骤与返回。

#### FR-8.2 可访问性基线自查 — P2 [B]

**需求**：发布前一轮基线自查（控件可访问名称、次级按钮对比度、长列表键盘遍历），记录已查项与已知缺口，不宣称 WCAG 合规。

## 7. 非功能需求

| 类别 | 要求 |
|---|---|
| 性能 | AI 操作首个可见反馈 ≤2s；非 AI 交互 ≤200ms |
| 可靠性 | AI 失败路径全部有文案+重试；兜底内容永远标注来源 |
| 数据 | 本地 SQLite + Drizzle；新模型见 §3.3；payload 带 `schema_version`；JSON 导出覆盖新模型 |
| 兼容 | **无旧数据兼容承诺**（FR-0.1）；提供重置与种子脚本 |
| 规则 | 平台规则注册表带来源/日期/版本（FR-0.2）；规则可能变化，实现记录核对日期 |
| 隐私 | 密钥只读环境变量；档案仅本地存储 |

## 8. 发布标准（Definition of Release）

v1.0 正式发布需全部满足：

1. M1 全部验收通过；M2 无 P1 遗留（P2 可豁免至 v1.0.x）。
2. **三平台真实后台演练**：用真实账号对 X（含 Thread）、小红书、公众号各完成一次「从 Muse 复制/上传到发布前一步」的演练并记录（评审 C §8/§9）。
3. 平台产物硬验收：
   - X：中文、emoji、URL 均按官方加权规则正确计数；Thread 逐条复制顺序正确。
   - 小红书：无图不得判定就绪；图片可排序并按序打包下载。
   - 公众号：导出 HTML、封面、摘要、正文图能实际带入公众号编辑流程。
4. 动线验收：修改通用稿不静默覆盖平台手工编辑；公众号未就绪时可单独发布已就绪的 X 作品。
5. AI 对抗性测试集全部通过（FR-4.1），派生/适配路径无静默新增事实断言。
6. 冷启动脚本：空库 → 三起点各建一个项目（含一个单平台直写、一个多平台派生）→ 走完发布与复盘 → 经验复用，全程无手动刷新、无矛盾状态。
7. `./init.sh` 全绿；640px 核心动线人工走查通过。

## 9. v1.x 展望（本版不做）

- 全 AI 操作流式化（初稿/派生渐显）。
- 本地内容日历/看板（计划日、多平台完成度聚合）。
- 个人数据基线（近 10 篇中位数对照）与实验卡（观察/假设/下次实验/验证状态，跨文章聚合验证）。
- 标签治理（噪声过滤、合并/隐藏/批量清理、AI 标签区分）。
- 小红书视频笔记、X Premium 长帖。
- 小红书图文卡片生成器（把要点渲染为可下载的图片卡片，补齐「图从哪来」）。
- 真实平台发布 API（`PublisherAdapter` 已预留）与表现数据自动抓取。
- 素材语义推荐（写作时推荐库内证据）。
